# 本地公式验证实验室设计

## 状态

已确认设计，等待用户审核本文档后进入实现计划。

## 背景

东方财富通验证副图数量持续增加，且长公式后段绘图存在兼容和空间限制。继续把所有诊断信息塞进东方财富通副图，会让界面拥挤，也会让“没显示”与“条件不成立”混在一起。

因此第一阶段改为建设一个本地公式验证实验室，用独立数据、独立计算和独立界面承载主图、副图、变量断点和中文原因链。它不是一开始替代东方财富通完整交易软件，而是先解决公式验证和诊断问题。

## 第一阶段范围

第一阶段选择方案 A：本地验证实验室。

不做：

- 不先做完整替代平台。
- 不先下载全A全历史。
- 不先做完整回测系统。
- 不改当前主公式。

先做：

- 读取少量样本股票。
- 下载前复权日线。
- 本地缓存数据。
- 计算基础指标和公式变量。
- 复刻一个验证链路，优先从 A3b2b1 开始。
- 在浏览器界面展示主图、副图和条件断点表。

## 数据来源

股票池来源：

- `D:\数据包.xlsx`
- 当前文件是 8 只股票的横截面/自选股快照。
- 可作为第一版股票池和当前横截面信息。
- 不能直接计算历史窗口指标。

历史行情来源：

- 第一版主数据源采用 AKShare。
- 接口采用 `stock_zh_a_hist`。
- 数据源口径为东方财富历史行情。
- 周期使用 `period="daily"`。
- 复权使用 `adjust="qfq"` 前复权。

Tushare 状态：

- 用户已提供 Tushare token，但不得明文写入文档、代码或日志。
- 当前 Tushare 积分为 100，不足以支撑前复权主数据源。
- Tushare 暂缓，后续积分足够后作为正式源或交叉校验源。

## 数据口径

第一版必须使用前复权数据，因为当前人工验证截图主要来自东方财富通“日线 / 前复权”界面。若使用未复权数据，BOLL、PCTB、二年位、突破背景和肉眼图形都会与当前验证场景不一致。

核心字段要求：

- 日期
- 股票代码
- 开盘
- 最高
- 最低
- 收盘
- 成交量
- 成交额
- 涨跌幅
- 换手率

字段单位需要在冒烟测试中确认并标准化，尤其是成交量、成交额和日期排序。

## 第一只冒烟测试

优先测试：

- 股票：`002222 福晶科技`
- 时间：`2024-01-01` 至当前日期
- 周期：日线
- 复权：前复权 `qfq`

冒烟测试要确认：

- AKShare 能正常安装和调用。
- 返回字段包含价格、成交量、成交额、涨跌幅、换手率。
- 日期顺序可稳定转换为升序。
- 前复权价格与东方财富通前复权界面大体一致。
- 数据能够缓存到本地文件。

## 指标与公式计算

第一版先实现公式验证所需的基础函数：

- `REF`
- `HHV`
- `LLV`
- `COUNT`
- `MA`
- `STD`
- `CROSS`

第一批指标：

- BOLL 中轨、上轨、下轨
- PCTB
- 带宽
- 均额
- 二年位

第一条公式诊断链：

- A3b2b1 单根上穿背景诊断

原因：

- 这是当前主线最新断点。
- 它已经被压缩成“只画成立项”的稳定思路。
- 它能验证平台是否可以替代东方财富通副图进行条件拆解。

## 界面设计

第一版界面采用本地浏览器页面。

主要区域：

- 左侧：股票池、股票搜索、日期选择、周期选择。
- 中间：主图K线和若干副图。
- 右侧：当前K线条件断点表和中文原因链。

第一版不追求复杂美化，重点是让诊断信息不拥挤：

- 主图显示前复权K线和基础指标。
- 副图可以按模块展开，不受东方财富通空间限制。
- 断点表显示每个变量成立/不成立。
- 中文解释说明当前卡在哪一层。

## 技术路线

推荐路线：Python 数据层 + React/Vite 前端。

Python 负责：

- 读取 Excel 股票池。
- 调用 AKShare 下载前复权日线。
- 缓存原始数据和标准化数据。
- 计算指标和公式变量。
- 输出前端可读取的 JSON/CSV。

React/Vite 负责：

- 股票和日期切换。
- 主图和副图展示。
- 断点表和原因链展示。

第一步先实现数据下载冒烟，不立即搭完整界面。

## 错误处理

数据下载失败时：

- 记录股票代码、日期范围、错误信息。
- 不中断整个任务。
- 支持重试。
- 支持跳过失败股票继续处理其他股票。

数据字段缺失时：

- 明确报出缺失字段。
- 不静默用空值替代核心价格字段。

复权口径不一致时：

- 标记为阻断问题。
- 不进入公式诊断。
