# 协作规范与公式优化方法

这份文件用于记录本次东方财富通公式讨论中形成的协作方法、优化思维和互动习惯。后续继续讨论主公式、验证副图、回测优化，或在其他对话中恢复协作方式时，可优先参考。

## 沟通方式

- 讨论复杂公式时，不要急着一次性推进太多内容。
- 每次只讨论一个小模块、一个变量或一组紧密相关条件。
- 对公式片段的讨论遵循固定流程：
  1. 先给出公式；
  2. 用中文解释每一句公式的含义；
  3. 评价当前写法是否合理；
  4. 主动指出潜在问题、不利影响或盲点；
  5. 给出1到2个可选优化方向；
  6. 最后让用户确认。
- 用户确认前，不把建议当成最终修改。
- 如果之前已经讨论过某项，但当前是在“重新从头讨论”，仍然要重新解释、评价、指出问题和优化方向，不要简单说“已确认过”然后跳过。
- 当用户说“一点一点来”“不着急”“重新讨论”时，必须放慢节奏，只推进当前最小单元。
- 允许用户随时插问。插问处理完后，要能回到原主线，并说明主线停在哪里。
- 如果用户指出推进太快、解释不清、概念混淆，助手要立即停下来，回到最近一个已确认点重新梳理。
- 对容易混淆的名称必须主动区分，例如 `蓝显示许可` 和 `显示许可` 不是一回事，`迟发挤压背景` 和 `有效挤压` 不是同一层级。

## 决策状态管理

- 每个结论都要有明确状态：
  - 已确认；
  - 暂时不改，后续验证；
  - 待讨论；
  - 已取消；
  - 只记录，不进入主公式。
- 对“暂时不改”的内容，必须同时记录后续要验证什么。
- 对“待讨论”的内容，后续继续时要主动带回来，例如黄信号后的 `确`。
- 对已确认修改，要同时记录影响和潜在副作用。
- 如果一个变量后来重新有用途，例如 `日线`，要更新之前“可删除”的判断。
- 当前公式结论、待办和验证项应持续写入 `formula-discussion-summary.md`。

## 公式解释习惯

- 任何涉及公式的回复，公式后面必须紧跟中文解释，不能只在公式前笼统说明，也不能假设用户能自行读懂公式。
- 中文解释必须贴着公式逐句或分组解释：每个关键变量、窗口、阈值、比较对象、冷却条件、显示语句都要说明白。
- 如果公式块超过 3 行，公式后至少要用项目符号逐条解释；如果公式只是 1-2 行，也要用一句中文说明它最终判断什么。
- 中文解释要说明：
  - 变量是什么；
  - 窗口是多少；
  - 当前K线、前一根K线、历史窗口分别指什么；
  - 条件成立代表什么市场含义；
  - 条件会影响哪些后续模块。
- 对比例和阈值要用直观语言解释，例如：
  - `带宽=0.25` 表示布林上下轨总宽度约为中轨价格的25%；
  - `1.08倍均宽` 表示当前带宽最多比平均带宽高8%；
  - `REF(...,1)` 表示排除今天，看上一根或今天之前。
- 对 `REF`、`COUNT`、`HHV`、`LLV`、`BARSLAST` 这类历史窗口函数，要明确说明是否包含当前K线、是否排除今天、窗口从哪根到哪根。
- 对动态阈值要解释其分档逻辑，例如高位更严、低位稍宽。
- 对公式用途要说明它属于哪一层：底层、背景、源头、过滤、确认、风险、最终显示。
- 每次讨论一个新变量、条件或模块时，开头必须先说明“它是用来干嘛的”，以及“最终是用来判断什么”。最低要求包括：
  - 当前对象：正在讨论的是底层变量、过滤条件、保护条件、风险信号、确认信号，还是显示条件；
  - 直接用途：它在当前组合里负责放行、拦截、降级、升级、冷却、互斥，还是只做观察；
  - 最终判断对象：它最终服务于判断买点质量、卖点风险、资金失衡、动能衰减、结构转弱、信号是否显示，还是信号显示优先级；
  - 进入位置：它是直接进主公式、进验证副图、只记录观察，还是暂不采用。
- 如果一个变量是“反向定义”或“保护条件”，必须特别说明它不是在直接证明信号成立，而是在排除误判。例如 `三日推进乏力` 是用来判断连续放额靠近前高时是否缺少推进，不是单纯判断涨幅大小。

## 公式修改对比格式

- 每个重大修改都要按固定格式整理：
  - 原公式；
  - 原公式中文解释；
  - 新公式；
  - 新公式中文解释；
  - 影响；
  - 潜在不利影响；
  - 后续验证点。
- 只说“改了什么”不够，必须说明“会影响哪些信号”和“可能带来什么副作用”。
- 如果用户要求对比，不能只列公式，要解释影响。
- 如果一项修改会让信号减少、增加、提前、延后、分层变化、风险压制变化，都要说清楚。

## 优化原则

- 不为了适配某一只股票、某一段历史走势而过度调整公式。
- 不追求“过去看起来特别准”而牺牲客观逻辑。
- 所有修改必须逻辑顺畅、合理、周延，且程度适中。
- 对个例问题，先判断是否代表一类稳定现象；若只是个例，不为其专门加复杂条件。
- 优先使用通用逻辑、分层提示、参数微调，而不是堆叠过多特殊条件。
- 没有验证或回测证明前，不盲目新增复杂条件。
- 优先做减法，优先保留能明显改善胜率、回撤、盈亏比或解释力的模块。
- 公式优化优先保证逻辑合理，再追求历史表现。
- 如果一个条件只是为了解决个例，但没有稳定市场含义，不应加入主公式。
- 如果一个条件有用但不应压制主信号，可以考虑降级为提示、验证字段或风险观察项。
- 参数调整要适中，尽量先小幅微调，并记录为什么选择这个幅度。
- 当一个模块已有后续宽度许可或风险过滤时，上游条件可以适度简化，避免重复过严。
- 不能形成“默认暂时不改、都等回测”的思维定势。每个问题都要客观判断：是否逻辑上现在就应该修改，是否适合先记录后验证，还是应该保持不动。
- 如果一个问题已经能从逻辑上明确存在，且修改通用、适中、不会显著增加复杂度，可以现在就优化，不必机械等待回测。
- 如果一个修改牵动范围很大、逻辑尚不确定或容易过拟合，则先记录为高优先级验证项，而不是草率修改。

## 主动性要求

- 不只是被动确认用户提出的想法。
- 助手需要主动发现可能的问题，例如：
  - 条件是否偏严或偏松；
  - 是否漏掉重要启动背景；
  - 是否错放已经扩张后的旧背景；
  - 风险是否过早或过晚；
  - 是否误伤强趋势；
  - 是否会产生过拟合；
  - 某个模块是否和另一个模块重复或冲突。
- 主动提出可能的优化方向，但必须让用户确认后才视为确定。
- 用户没有想到的潜在问题，助手也要主动加入后续验证和回测清单。
- 助手提出的所有“可能问题”也必须负责到底，后续进入验证和回测清单。
- 助手要主动关注模块之间是否重复、冲突或职责不清，例如迟发挤压只做突破背景，有效挤压才进入真正收缩。
- 当用户提出某个新条件时，助手要主动判断它更适合放在变量本身、合成条件、显示层、风险提示还是验证副图里。
- 助手也要主动帮助用户更充分地发挥助手能力。用户不需要天然知道如何“用好助手”，助手应在合适时提醒更高效、更全面、更稳健的协作方式。
- 如果某个任务更适合先建立框架、表格、清单、验证方案、对比模板或记录文件，助手应主动提出。
- 如果用户的问题容易陷入局部个例，助手应主动提醒回到通用逻辑、样本验证和分层判断。
- 如果讨论内容已经复杂到容易遗忘上下文，助手应主动建议阶段总结、更新记录或建立待办清单。
- 如果可以通过验证副图、回测脚本、分层统计、案例对照等方式提高判断质量，助手应主动提出，而不是只回答当前问题。
- 助手要在协作过程中不断总结更有效的方法，并把有效方法固化到本文件中，让后续协作越来越强。
- 用户不需要记住助手能做的全部事情。助手应主动判断何时使用项目文件、版本管理、对比表、验证副图设计、Python回测脚本、错误定位工具、公式整理重构、阶段总结等能力，并在合适时提醒用户。

## 主动盲点扫描

- 在评价任何公式条件时，助手不能只解释它“正常情况下怎么工作”，还要主动扫描它可能遗漏、误放、重复、过严、过松的边界情况。
- 每次讨论一个条件时，助手应主动问自己：
  - 这个条件是否只看了次数，没看幅度；
  - 是否只看了幅度，没看过程；
  - 是否只看了结果，没看所处位置；
  - 是否只看了当前K线，忽略了启动前状态；
  - 是否可能被单日极端值污染；
  - 是否存在 OR 条件过宽或 AND 条件过严；
  - 是否和其他模块重复限制；
  - 是否可能误伤强趋势二次机会；
  - 是否可能把旧背景当成新背景；
  - 是否需要放进验证副图，而不是直接改主公式。
- 对固定窗口、固定阈值、固定倍数，助手还应主动考虑是否需要自适应参考：
  - 是否应根据日线、周线、月线不同周期自适应；
  - 是否应根据个股波动状态自适应；
  - 是否应根据价格所处历史分位自适应；
  - 是否应根据近期参考与长期参考差异自适应；
  - 是否应根据趋势阶段、压缩阶段、高宽阶段自适应。
- 自适应不是默认要加入主公式。助手需要判断：当前固定参数是否已经足够，是否只需进入验证副图观察，还是逻辑上已经有必要做动态分档。
- 当用户指出一个重要盲点时，助手必须反向提炼为通用检查项，而不是只修补当前问题。
- 对公式模块的回答里，除非问题非常简单，否则应包含一个“主动盲点”或“需要特别验证”的判断。
- 如果助手发现自己是在用户提醒后才意识到重要情况，应立即补充到待验证清单或协作规范中，防止同类遗漏重复出现。

## 触发词自动收录机制

只要讨论中出现“未来要判断、当前暂缓、可能有副作用、会影响某个模块、可能要改条件、需要分层或自适应”的表达，助手必须当场判断是否写入 `pending-verification-list.md`。

按 6 类触发器执行：

1. 验证类
   - 后续验证、后续观察、回测重点、验证副图、进入副图、后面看、需要观察、统计一下、对比一下、后续测试、专项测试。

2. 风险/问题类
   - 可能偏宽、可能偏严、可能偏慢、可能过早、可能过晚、可能误伤、可能漏掉、可能噪音、可能重复、可能冲突、可能冗余、可能失效、可能错放、可能压制、可能放过、可能不稳定。

3. 候选/暂缓类
   - 候选优化、候选逻辑、先不改、暂不加入主公式、暂时保留、暂时接受、先记录、先观察、以后再看、等回测、等验证、后续再决定。

4. 条件变化类
   - 放宽、收紧、删除、保留、简化、替换、合并、拆分、降级为提示、升级为信号、加入主公式、不进主公式、只做提示、只做验证。

5. 模块影响类
   - 会影响温上/暴上/极上、会影响蓝/黄、会影响启/冲/速、会影响确、会影响险/败/慎/滞、会影响显示层、会影响冷却、会影响突破背景、会影响真正收缩、会影响风险过滤、会影响信号数量。

6. 自适应/分层类
   - 是否需要自适应、是否要按周期分档、是否要按二年位分档、是否要按高位/低位区别、是否要按趋势阶段区别、是否要按带宽状态区别、是否要按量能分位区别。

执行规则：

- 每一小块结束时，助手必须做一次收录检查：是否新增待验证项、验证副图变量、候选逻辑。
- 如果有新增项，助手要明确说明“已记录”或“现在记录”。
- 如果不记录，要说明原因，例如已经被某个现有验证项覆盖。
- 一个模块里有多个验证点时，不能只记录其中一个；要按模块拆开记录。
- 每完成一个大模块，要做一次摘要一致性校验，检查确认过的改动、候选项、验证项是否进入对应文件。

## 助手能力触发时机

- 当讨论产生多个确认结论时，主动更新讨论摘要或项目文件。
- 当修改足够大时，主动建议写入 `formula-changelog.md`。
- 当存在多个版本或阶段时，主动进行版本化管理，例如原版、讨论版、确认版、待回测版、回测后优化版。
- 当用户要求或讨论涉及“原来/现在/影响”时，主动生成对比表或对比结构。
- 当信号原因难以肉眼判断时，主动建议进入验证副图或条件断点表设计。
- 当用户准备导出数据或讨论统计效果时，主动建议 Python 回测脚本和统计口径。
- 当公式变量重复、顺序混乱、无用变量较多时，主动建议整理和重构。
- 当对话变长、上下文复杂、已确认项变多时，主动进行阶段总结和恢复上下文。
- 当出现“为什么这里出/不出信号”这类问题时，主动建议使用诊断字段、断点表或验证副图，而不是从头逐条人工排查。

## 上下文管理机制

- 长对话不能只依赖聊天记录。助手应主动把上下文项目化，使用项目文件保存结论、断点、版本和待验证项。
- `current-checkpoint.md` 专门记录当前主线、最近确认项、当前待确认项、插问处理状态和下一步。
- 每完成一个大模块、一次较长插问、一次重要方法改进，或出现“我们讨论到哪里了”的风险时，助手应主动更新 `current-checkpoint.md`。
- 如果上下文变长但助手没有主动提醒或维护断点，属于主动性不足，后续需要立即修正。
- 恢复讨论时，优先查看 `current-checkpoint.md` 和 `formula-discussion-summary.md`，避免把旧版本结论和新版本结论混在一起。
- 如果发现项目文件与最新讨论不一致，助手应主动提出做“摘要一致性校验”，而不是继续带着冲突推进。

## Token 节省与上下文压缩

- 在不影响沟通质量和公式理解的前提下，助手应主动减少重复上下文消耗。
- 节省 token 的核心方式不是省略关键解释，而是把长期上下文压缩成可引用文件和短状态。
- 对已确认结论，后续优先引用 `formula-discussion-summary.md` 或 `current-checkpoint.md`，不反复完整复述全部历史。
- 对已作废方案，助手应主动标记为作废，避免后续继续在旧分支之间比较。
- 当用户插问结束后，助手用简短断点恢复主线，不重新铺开全部历史。
- 当讨论进入新模块时，助手只带入必要前置条件；除非用户要求对比，否则不重复展开远处模块。
- 当内容过长时，助手应主动建议或执行阶段压缩，格式包括：当前主线、已确认、待继续、待验证、作废内容、下一步。
- 长材料、公式版本、验证清单、回测口径应尽量放入文件，由助手按需读取，而不是每轮复制到聊天里。
- 节省 token 不能牺牲以下内容：
  - 涉及公式时必须中文解释；
  - 重大修改必须说明影响和潜在副作用；
  - 待验证项必须记录；
  - 用户要求“一点一点来”时不能为了省 token 跳步。
- 如果助手发现自己在重复解释已经确认且无新增信息的内容，应主动压缩表达，并说明“这里引用已确认结论，不重复展开”。

## 引用内容可追溯

- 当用户通过选中文本、截图或“添加到对话”来追问某一段内容时，助手应先简短复述被追问的关键引用，再回答问题。
- 如果界面没有把用户选中的原文显式显示在对话记录里，助手要主动补充“你问的是这句/这段：……”以便后续回溯。
- 复述引用时只保留必要片段，避免大量重复，但必须足够让用户以后看聊天记录时知道当时在讨论哪句话。
- 如果用户追问的是公式片段，助手应同时保留原公式片段，并继续遵守“公式后必须中文解释”的规则。
- 如果用户的引用不完整或上下文可能有歧义，助手要先说明自己的理解，再回答。

## 高质量协作模式提醒

- 助手应主动把有效的 AI 协作方法转化为本项目可执行的流程，而不是只停留在口头建议。
- 如果某个任务适合写入项目级说明文件，例如 `AGENTS.md`、协作规范、公式说明或回测说明，助手应主动提醒用户：这里可以沉淀成项目说明，方便后续对话和版本继续使用。
- 当进入复杂公式整理、重大逻辑重构、版本定稿或回测前检查时，助手应主动建议使用 Writer / Reviewer 模式：
  - Writer 模式负责整理公式、实现修改、生成版本；
  - Reviewer 模式负责重新审查逻辑漏洞、重复变量、兼容性问题、过拟合风险和遗漏验证项。
- 每个实质性公式修改都必须绑定验证方法。助手不能只说明“这样更合理”，还要说明后续应通过验证副图、条件断点表、案例对照或 Python 回测观察什么。
- 如果助手已经在执行某种优质协作模式，可以不额外打断用户说明；如果尚未执行，而当前场景适合使用该模式，助手应主动提醒用户。

- 当连续讨论已经积累出多项方法改进时，助手应主动更新本文件，使协作方式持续进化。

## 可以进行的协作方式

这些方式用于把“助手能做什么”转成日常协作中的具体动作。用户不需要提前知道或点名这些方式，助手应根据场景主动判断是否适合使用。

- 日常讨论：用户只提出方向、疑问或插问时，按“一次一个最小单元”推进，先讲中文逻辑，再必要时写公式，最后让用户确认。
- 公式完善：适合按“用途定位 -> 公式 -> 中文解释 -> 合理性评价 -> 盲点/副作用 -> 1到2个优化方向 -> 用户确认”的流程协作。
- 验证副图同步整理：讨论每个公式小模块时，同步整理验证副图需要显示的变量、窗口、阈值、成立状态、过滤/冷却/互斥影响、显示文字和后续验证点。
- 替代信号主动提示：讨论某一层、某个模块或某类信号时，助手必须主动告知还有哪些其他可用信号或指标、它们是否可能更好、是否与现有信号重复、是否应进入验证副图或主公式。
- 诊断排查：遇到某根K线为什么出信号或不出信号时，优先使用诊断链、验证副图或条件断点表，不凭感觉猜原因。
- 文档沉淀：当讨论产生已确认结论、待验证项、版本变化、当前断点或协作方法变化时，主动写入对应项目文件。
- 版本整理：进入公式整理版、定稿、回测前检查或重大重构时，可使用 Writer / Reviewer 分工，先生成版本，再独立审查漏洞。
- 数据和回测：当需要判断参数质量、信号副作用、误伤、漏报或不同方案优劣时，主动建议统计、对比或脚本回测。
- 长上下文管理：对话变长、插问结束或主线容易丢失时，主动更新断点并恢复主线；必要时建议新开对话和准备交接文件。
- 能力提醒：当某个问题明显适合使用项目文件、对比表、验证副图、脚本、复盘或审查流程时，助手应简短提醒可用方式，但不能打断当前最小讨论单元。

执行要求：

- 验证副图信息可以先以“字段草稿”形式记录，不要求每次讨论都立刻写完整副图公式。
- 已确认字段、待确认字段、候选字段要分开标注，避免副图口径提前定死。
- 每一小块结束时，除讨论结论外，还要做一次“验证副图字段检查”。
- 每一类信号确认前，要补充一次“替代信号检查”：是否存在同类指标、是否更适合作为主公式、是否只适合验证、是否应暂不采用。
- 不要习惯性把所有新想法都放入验证。若某个优化逻辑清楚、通用性强、复杂度适中、与系统分层职责一致且副作用可控，应直接告诉用户它适合吸收进主公式；验证只用于后续观察效果和参数微调。
- 验证工作量必须被主动控制。讨论中出现新想法时，助手要先判断它属于“可直接改”“需要验证”“暂不采用”哪一类；不能把可直接优化的问题、明显重复的问题、价值不高的问题都塞进验证清单，导致后续验证负担过大。
- 每个条件定稿前，必须做“同类参照补全检查”。不能只沿着当前公式已有变量微调，要主动问：这个条件除了和长期均值比，是否还应和近期均值、前一根、短期前高/前低当天、阶段峰值、分位水平或同层替代指标比较；若某个参照更贴近该条件的市场含义，应主动提出并判断是直接进主公式、进入验证，还是暂不采用。
- 用户要求“一点一点来”时，仍然要完成当前最小单元内的主动补全；慢节奏不是只被动回答当前变量，而是每次只处理一个小模块，并把这个小模块的关键参照关系补完整。
- 每个模块或关键条件开始讨论前，必须做“第一性原理检查”。先用一句话说明这个条件本质上要识别什么市场现象，再拆出该现象最直接的可观察证据、最关键的比较对象、最容易误判的边界，最后再落到公式变量。不能先被原公式变量锚定，再局部修补。
- 第一性原理检查至少回答：这个信号想证明什么；如果它真的发生，K线/价格/量额/指标应留下什么痕迹；最自然应该和谁比较；什么情况看起来像但其实不是；该条件应直接进主公式、进验证副图还是暂不采用。
- 第一性原理不能只在模块开头说一次，定稿前必须再执行“本质命题复核”。固定五步为：1. 用一句话重述本质命题；2. 列出必须同时成立的核心证据；3. 区分基础条件、对照条件、加强条件和排除条件；4. 对 OR/AND 合成做反例压力测试；5. 决定直接进主公式、进验证副图或暂不采用。
- 凡是公式里出现多个 OR 条件，必须额外检查是否把“任一迹象”误写成了“充分证据”。如果信号本质需要多类证据共同证明，应优先考虑 `基础条件 AND 对照条件` 的结构，而不是简单 OR。
- 凡是准备保留或提出固定阈值、固定窗口、固定倍数前，必须做“阈值严松影响复核”。至少说明：该阈值对应的直观含义；偏严会漏掉什么；偏松会误放什么；是否可用另一个条件配合来放宽阈值但控制噪音；该阈值应直接改、进入验证还是暂时保留。
- 提出阈值组合时，必须做“临界样本反推”。主动构造刚好触发的边界例子，检查它是否符合信号本质。例如 `收盘位置<0.6 AND 上影占比>0.35` 要反推 `收盘位置=0.59、上影占比=0.35` 是否真的应算冲高回落；若边界样本不符合直觉，应主动收紧或重构条件。
- 凡是条件只看区间涨跌幅、振幅、累计量额、次数或最终结果时，必须做“过程结构检查”。主动问：过程中是否在持续抬高、持续走弱、低点抬高、高点抬高、收盘抬高、重心上移或重心下移；不能把温和推进误判成乏力，也不能把表面不跌误判成健康。
- 凡是讨论均线、布林中轨、平台线、前高前低、趋势线等结构类条件时，必须同时检查“位置”和“方向”。不能只看价格是否跌破、贴近或站上线，还要主动检查方向变化，例如均线是否下压/上拐、斜率是否走平、价格是否回不去、平台低点是否下移、收盘重心是否持续变弱。结构转弱类信号尤其不能只凭“碰线/破线”定性。
- 判断“乏力、推不动、滞涨、横盘”等状态时，必须额外检查“最终有效上推”和“先弱后强”。如果窗口内曾微跌或震荡，但最后已经有效拉升、突破或收盘打开空间，不应简单归为乏力；应先判断是否需要排除、降级或转入其他强势状态。
- 凡是讨论两个可能重叠的信号，必须先做“互斥边界推演”，再讨论显示优先级。至少要问：这两个信号分别想证明什么；什么情况应该明确归 A 而不是 B；有没有一种边界样本会同时满足但本质只属于其中一个；公式窗口是否把今天和旧参照混在一起；信号名称是否暗含“新高、失败、突破、滞住”等方向性含义。
- 凡是某个信号要作为后续确认、失败或跌破中位的基准 K，必须先判断它是单日事件还是区间状态。单日信号可以直接用显示日作为基准 K；区间状态不能默认用显示日作为基准 K，必须先定义代表 K，例如区间最大成交额、区间最高价、区间最明显冲高回落，或暂不纳入确认信号。
- 区间状态选择代表 K 时，必须先回到该信号的核心证据，代表 K 应优先对应“最能代表本质的那一根”。例如资金失衡/连续放额类信号，优先考虑区间最大成交额 K；冲击前高类信号，优先考虑区间最高价 K；冲高失败类信号，优先考虑上影/回落最明显 K；不能随手使用显示日。
- 每进入一个新层级前，必须做“层级完整性扫描”。先从该层本质出发列出所有主要表现方向，不能只列当前想到的几个模块。例如资金失衡层应同时扫描：放量不涨、放量冲高回落、放量下跌、连续放量推不动、单日极端放量、放量破位等方向。每个方向都要判断：进入主公式、进入验证副图、归入其他层级、合并到已有模块或暂不采用。
- 层级完整性扫描必须特别检查“相反方向”和“更严重形态”：如果列了放量滞涨，就要主动问是否还有放量下跌；如果列了缩量背离，就要问是否还有放量失败；如果列了预警，就要问是否有确认或升级版本。

## 分层思维

- 区分不同层级：
  - 底层数据：K线、量能、带宽、PCTB、分位；
  - 背景：近挤、迟发挤压、有效挤压、真正收缩；
  - 源头：温上、暴上、首扩、速、冲、平台；
  - 质量过滤：宽度许可、压缩质量、量价质量；
  - 分层显示：蓝、黄、启、速、冲、确；
  - 风险提示：险、败、警、慎、滞；
  - 最终互斥：哪个信号优先显示。
- 不同层级的条件不能随意混用。
- 一个条件“可以作为突破背景”不代表它“可以进入有效挤压”。
- 一个条件“可以解释风险”不代表它必须压掉启动信号。
- 一个条件“可以用于验证副图”不代表它必须进入主公式。

- 不仅记录当前和未来讨论的问题，也要回溯记录以前已经讨论过的问题。

- 后续验证和回测阶段要逐项检查：
  - 是否真的存在该问题；
  - 是逻辑问题还是参数问题；
  - 应该放宽、收紧、拆分、降级为提示，还是保持不动；
  - 修改后是否带来新的副作用。

  - `formula-discussion-summary.md`：公式结论、变量、待验证问题；

## 验证副图思路

- 后续当用户说“开始验证指标讨论”后，单独设计一套验证和诊断副图指标。
- 验证副图不是交易信号本身，而是解释器和公式校准仪表盘。
- 它要帮助判断：
  - 为什么某根K线出信号；
  - 为什么某根K线没出信号；
  - 这里出信号是否合理；
  - 这里没出信号是否合理；
  - 哪个条件或阈值需要调整。
- 验证副图必须覆盖核心判断链，包括但不限于：
  - 基础K线：开高低收、实体、实体占比、上影、下影、阳强、长上；
  - 量能：成交额、均额、有额、温额、放额、巨额、缩额；
  - 布林/PCTB：MID、BUP、BDN、PCTB、有效上穿、确认强区、布林滞后保护；
  - 带宽：带宽、有效宽均、有效高宽、当前宽比、前宽比、宽近高、宽速率、持宽；
  - 分位与动态阈值：二年位、高位2年、低位2年、迟发绝宽线、低位均比、低位低比等；
  - 挤压：挤压、近挤、效挤、分散挤原、挤压宽许可、有效挤压、迟发挤压背景；
  - 收缩：压缩低位、收缩过程、真正收缩；
  - 趋势结构：构上、构下、短线、长线、短向、长向、强多、初多、上轨趋势；
  - 启动与分层：启、蓝、黄、冲、速、平台、优质黄背景、蓝压缩质量；
  - 确认/风险/失败：确2、确3、确4、险反、险、败、警、慎、滞。
- 验证副图要显示动态阈值本身，不只显示条件结果。例如不仅显示是否通过 `迟发绝宽许可`，还要显示当时的 `迟发绝宽线`。
- 验证副图要能判断“没出信号是否合理”和“出了信号是否合理”，而不只是解释公式为何成立。
- 验证副图要服务于发现公式缺陷，包括漏掉好背景、错放坏背景、风险漏报、风险误杀、信号优先级错误等。
- 验证副图的目的不仅是解释信号，还要帮助反向优化主公式。
- 设计验证副图第 1 区时，基础层口径必须固定为“基础K线 + 价格位置 + 成交额/量能 + 相对异常程度”。不能只把基础层理解为K线形态。
- 成交额是横跨温上、暴上、首扩、速、冲、确、险、慎、滞的底层变量。即使当前讨论的是形态区，也必须主动检查是否需要同步纳入量能字段。
- 诊断时只要有一个决定性变量缺失，就要立即判定当前验证副图对该问题信息不足，不能继续凭感觉硬查。
- 决定性变量是指：没有它，就无法判断某个关键条件为什么成立或失败。辅助变量缺失可以先继续，但必须记录为待补字段。
- 信息不足时，应先归类缺口，例如基础形态、成交额、价格位置、带宽、挤压/迟发、收缩、源头、显示/过滤、风险，再补字段。

## 回测流程

- 主公式逻辑先讨论完善。
- 然后设计验证副图，人工检查公式是否合理。
- 再根据验证结果做逻辑和参数微调。
- 公式结构稳定后，再做实际回测。
- 回测时分层统计：
  - 启动类：启、蓝、黄、速、冲；
  - 确认类：确2、确3、确4；
  - 风险类：险、败、警、慎、滞；
  - 组合类：启动后有确 vs 没有确，启动后出现风险 vs 没出现风险。
- 回测前先不要为了让图形好看而调整公式。
- 回测后也不能只看胜率，要同时看信号数量、最大回撤、最大涨幅、盈亏比、信号后确认效果、风险提示效果。
- 对确认类信号，要比较“启动后有确”和“启动后无确”的差异。
- 对风险类信号，要重点看其后最大回撤和继续上涨概率，而不是简单看涨跌。
- 回测后再根据统计结果做第二轮优化。
- 新公式整理完成后，必须与初始公式逐信号比较：
  - 不只比较启动/买入类质量信号，也要比较示警、危险、失败、谨慎、滞涨和中性提示信号；
  - 看新公式新增的启动信号是否有质量；
  - 看初始公式显示但新公式未显示的启动信号，是否有质量且被误伤；
  - 看新公式是否漏掉初始公式有效风险提示；
  - 看新公式新增风险提示是更及时，还是过度示警；
  - 对差异信号做原因归因，例如挤压背景变化、温上收紧、承接弱过滤、宽度许可、冷却、风险过滤或显示优先级变化；
  - 后续回测脚本应支持 `新旧同显`、`仅新公式`、`仅初始公式`、`均不显示` 四类标签。
- 后续验证/回测还必须做趋势识别校准：
  - 对照股票实际走势，判断向上趋势、向下趋势、整理后再启动、顶部风险或弱化趋势是否被公式识别；
  - 对漏识别趋势和误识别趋势分别归因；
  - 不能为了个别小众走势过拟合；
  - 如果某类趋势逻辑客观、条件适中，且不会明显伤害其他股票捕捉能力，应考虑尽量捕捉；
  - 对个例先判断是否代表稳定的一类现象，再决定加入主公式、降级提示、进入验证副图或保持不动。
- 后续验证/回测还必须判断当前公式的适配边界：
  - 如果某类走势客观上不适合当前公式捕捉，不要硬塞进当前公式；
  - 需要提醒用户这类走势更适合用什么其他指标原理识别；
  - 可针对这些原理另行设计新公式、辅助副图或提示指标；
  - 核心顺序是：先判断是否适合当前公式；适合则尽力捕捉，不适合则分流到新公式/新指标。
- “不为少数特殊形态扭曲主公式”的思维适用于所有模块：
  - 如果为了捕捉少数高波动、小众、特殊走势，会让普通样本条件过宽、过严或职责混乱，不应全局修改主公式；
  - 应先判断该情形是否能客观分层，例如按波动率、带宽分位、价格位置、量能分位、趋势阶段、周期类别；
  - 最优方案是公式能自动识别具体股票状态并分层处理：不伤害普通股票判断，同时增加对特殊股票如高波动股票的捕捉能力；
  - 能稳定分层且不伤害其他样本时，可以考虑分层逻辑；
  - 如果做不到稳定、客观、可解释的分层，优先满足大部分股票的判断；
  - 不能稳定分层时，优先作为验证提示、辅助公式、新公式方向，或保持不捕捉；
  - 这个原则适用于平台振幅、挤压背景、温上/暴上、首扩/续扩、风险、冷却、趋势确认等所有部分。

## 其他对话中的使用方式

- 如果新的对话没有继承本上下文，应先读取或粘贴本文件。
- 在其他对话中继续公式工作时，应先说明：“请按 `collaboration-guidelines.md` 的协作方式来讨论。”
- 若上下文丢失，本文件可作为恢复工作方式的基准。

## 持续迭代

- 本协作规范不是定稿。
- 随着后续公式讨论、验证副图设计、回测优化和用户反馈，助手需要继续更新本文件。
- 如果发现新的高效讨论方式、记录方式、验证方式或风险控制原则，应及时补充。
- 如果发现当前协作方式有问题，例如推进太快、解释不够清楚、记录不够完整、主动性不足，也应修正。

## 直接优化与后续验证的区分

- 不要习惯性把所有问题都留到后续验证。
- 如果问题属于变量命名错误、明显重复条件、逻辑绕圈、候选未通过却占用冷却这类“逻辑上明显更干净”的问题，可以直接优化。
- 验证主要留给阈值严松、边界权衡、不同股票适配、信号质量变化、潜在副作用这类不确定问题。
- 每次提出“先不改、后续验证”前，要先判断：这是不是可以直接优化的问题。
- 直接优化后仍可保留影响验证项，但不应因此拖延明显合理的结构修正。

## 全量逻辑复查机制

- 当出现“A OR A的宽松版”“严格条件被后续宽松条件绕开”“源头条件和保护过滤混在一起”“同一保护多层重复过滤”“候选信号未最终有效却参与冷却/新鲜度”等结构时，助手必须先做逻辑复查，而不是简单写入后续验证。
- 这类问题优先判断是否属于结构漏洞；如果是，先提出整理版直接优化方案，再把影响作为后续验证项。
- 凡是某个变量会进入“冷却、占用、新鲜度、趋势段管理、互斥压制”时，必须额外做“占用对象检查”：问清楚应由候选信号占用，还是由最终有效信号占用，还是由有效后的一段趋势状态占用。
- 固定提问句：`这个冷却/占用到底在管理谁？候选、最终有效信号，还是有效后的一段趋势状态？`
- 如果用户提出“通过有效后的冷却天数、后续股价趋势、重新收缩再扩张来限制”，助手应优先识别为结构更干净的“有效后段落管理”，不要继续默认用候选信号占用冷却。
- 占用对象检查至少回答：
  1. 这个冷却/占用到底在防什么；
  2. 候选占用会不会把未真正有效的噪声也记账；
  3. 只用最终有效占用会不会放过连续候选反复试错；
  4. 是否应该拆成“硬冷却 + 同段趋势禁止 + 回落/收缩/时间重置后允许”；
  5. 这属于结构修正还是阈值验证。
- 全量复查分三次进行：
  1. 当前讨论中发现一个模块存在类似风险时，立即局部复查；
  2. 主公式逐块讨论完成后，做一次跨模块结构审计；
  3. 整理版公式生成后，用 Reviewer 模式重新检查重复条件、绕路条件、显示空窗、动态引用兼容和旧变量残留。
- 真正的历史数据回测放在结构审计之后进行。先修明显逻辑结构，再用回测判断阈值、边界和信号质量。

## 条件分工表

- 对复杂变量，助手应说明它回答的是哪个问题，而不是只解释字面条件。
- 逐块讨论公式时，助手必须先说明该小块的用途：
  - 这一块准备用来干嘛；
  - 它直接判断什么；
  - 它最终影响哪个信号、过滤、保护或显示；
  - 它属于源头、背景、质量过滤、风险保护、风险出口、确认还是显示层。
- 说明完用途后，再展示公式和中文解释。这样用户可以先理解讨论目标，再提出更准确的问题。
- 对每个小块的评价必须分清四层：
  - 逻辑上是否合理：这个条件表达的市场含义是否顺；
  - 结构上是否干净：有没有重复、绕开、职责混淆、源头和过滤混在一起；
  - 是否可以直接优化：若是命名、重复、绕路、候选占用、保护误吞源头等结构问题，应优先提出直接优化；
  - 哪些需要验证：只有阈值严松、窗口长度、信号质量和副作用才主要交给验证/回测。
- 例如带宽/收缩相关变量可按以下分工解释：
  - 长期位置参考：回答“带口相对长期低不低”。
  - 中期高宽窗口：回答“前面有没有宽过”。
  - 近期低宽/低点窗口：回答“最近有没有收下来”。
  - 收缩幅度：回答“有没有从宽变窄”。
  - 低点新鲜度：回答“扩张前一刻是不是刚刚窄过”。
  - 第一段扩张限制：回答“是不是老收缩被反复复用”。
  - 高热/已走出限制：回答“它是不是已经不是初扩，而是进入加速后段”。
  - 阴线大冲回限制：回答“这根K是不是已经冲高失败，不应再被保护”。
- 同一个底层现象用于不同目的时，阈值可以分层：普通买点可以包容，优质信号更严格，凡是会挡掉风险信号的保护条件必须最严格。

## 本质命题复核五步

用户确认 `入规`：后续讨论每个公式模块时，不能一上来只给结论或公式。助手要引导用户从第一性原理进入，并按“本质命题复核五步”复核这个模块为什么存在、它想证明什么、证据如何分工、条件如何合成、最终落到哪里。

固定五步：

1. 重述本质命题：这个信号到底想证明什么？
2. 列核心证据：哪些证据必须同时成立，哪些只是辅助？
3. 区分条件分工：哪些是基础条件、对照条件、加强条件、排除条件？
4. 检查合成方式：OR 是否过宽？AND 是否过严？是否需要分层？
5. 判断落点：直接进主公式、进验证副图，还是暂不采用。

执行要求：

- 这五步不是形式化清单，而是模块讨论的入口。助手可以根据模块复杂度压缩表达，但不能跳过核心思考。
- 对用户不熟悉的模块，助手应先用市场现象语言解释，再落到公式变量。
- “最直接证据”优先于二手、绕路、复合很重的条件；如果必须使用复合条件，要说明为什么直接证据不足。
- “自然比较对象”必须主动寻找，包括前一根、最近几根、阶段高低点、历史均值、同类信号、趋势阶段、量能基准、波动基准等。
- “看起来像但其实不是”要主动列出排除对象，避免把相似走势误纳入主公式。
- 每个候选变量都要分流：进主公式、进验证副图、只记录观察、不采用；不能把所有想法都堆进主公式，也不能把明确关键证据漏出验证副图。
- 这个开场流程服务于“一点一点来”，不是加快跳步。讨论仍然每次只推进一个小模块。

## 外部沟通记录吸收原则

- 外部沟通记录只能作为参考材料，不作为当前公式的新主线。
- 吸收外部记录时，先区分三类内容：
  - 可吸收的方法：诊断流程、显示层分层、验证工具、记录方式；
  - 可重新审查的公式观点：风险分层、显示优先级、兼容性问题；
  - 不直接吸收的内容：旧对话里的跳跃式修改、个案化参数、未经当前讨论确认的结论。
- 阅读外部记录后，必须把有价值的内容转成当前项目体系里的条目，例如协作规范、验证副图计划、待验证清单、当前断点，而不是让旧记录直接支配当前讨论。
- 对旧记录中的公式建议，必须重新按当前协作流程讨论：先解释原逻辑，再评价合理性，再提出风险和优化方向，最后由用户确认。
- 如果外部记录暴露出“验证太久、先猜后查、思路发散”的问题，后续遇到单根K线诊断时必须先走固定诊断链，避免再次无序排查。

## 单根K线信号诊断链

- 遇到“这里为什么出信号”或“这里为什么没出信号”时，先按层级定位，不先猜原因。
- 推荐顺序：
  1. 源头层：先看 `首扩启`、`蓝温源`、`蓝暴源`、`黄温源`、`黄暴源`、`速有效`、`冲候选预`、`确`、`险原`、`慎源`、`滞源` 等是否成立。
  2. 有效层：再看 `蓝温有效`、`蓝暴有效`、`黄温有效`、`黄暴有效`、`冲有效`、`速有效` 等是否被冷却、趋势段、新鲜度、速后运行等过滤。
  3. 风险过滤层：检查 `险`、`上败`、`警源`、`初扩早期险过滤`、`扩张保护期`、`中位突破保护` 等是否改变最终显示。
  4. 显示优先级层：检查 `首扩显`、`蓝暴最终`、`速显`、`蓝温显`、`冲显`、`黄暴显`、`黄温显`、`确显`、`警显`、`慎显`、`滞显` 的互斥顺序。
  5. 兼容与版本层：检查是否存在动态 `REF`、变量重复定义、公式版本未同步、诊断变量残留、绘图覆盖等非逻辑问题。
- 正式主公式不要混入大量临时诊断代码。复杂定位应优先通过验证副图或条件断点表完成。
- 最终显示层尽量使用 0/1 归一化标记，例如 `险标:=IF(险,1,0);`，再用 `险标=0` 控制显示，减少公式系统对 `NOT(变量)` 的兼容风险。
- 最终显示互斥应优先使用“显示候选”而不是原始信号。也就是说，先让每个信号通过自身冷却/过滤形成可显示候选，再按优先级互斥；冷却中的高优先级信号不应继续压制新鲜的低级信号，避免主图没有任何提示。

## 重点清单与母清单完整性

- 当用户要求“重点内容”“全部内容”“完整清单”“后续必须验证内容”时，助手不能只凭聊天记忆回答。
- 回答前必须先说明口径：这是代表项、阶段重点，还是完整母清单。
- 如果声称是完整清单，必须先对照项目文件，至少包括：
  - `pending-verification-list.md`
  - `formula-changelog.md`
  - `formula-v1-change-notes.md`
  - `formula-discussion-summary.md`
  - `current-checkpoint.md`
- 更重要的是，`pending-verification-list.md` 本身也不能被默认视为永远完整。它是当前母清单，但需要定期做“母清单完整性审计”。
- 母清单完整性审计应至少覆盖：
  - 当前公式所有模块是否都有对应验证项；
  - 已确认直接优化项是否仍需后续效果验证；
  - 已暂缓项、候选项、用户插问项是否都落入清单；
  - 外部文档中被吸收的方法或问题是否已转化为本项目条目；
  - 验证副图是否覆盖判断信号成立/不成立所需的核心变量；
  - 新旧公式差异是否覆盖质量信号、风险信号和中性提示信号；
  - 是否存在公式职责之外、需要另做辅助公式或其他指标原理的事项。
- 如果发现母清单有遗漏，应先补清单，再基于清单继续总结。不能用不完整母清单去证明总结完整。
- 各个 MD 文件本身也必须尽量全面覆盖沟通过程中的有效信息。不能只收录被明确称为“重点”的内容。
- 只要沟通内容实际影响公式逻辑、参数判断、风险判断、验证计划、回测计划、验证副图字段、协作方式、命名规范、版本差异或未来决策，即使用户或助手没有使用“重点”这个词，也应判断是否写入相应文件。
- 文件分工原则：
  - `formula-discussion-summary.md`：记录已经讨论过的公式逻辑、解释、接受/暂缓/不采用的结论。
  - `pending-verification-list.md`：记录所有后续需要验证、回测、观察、专项比较、验证副图辅助判断的事项。
  - `formula-changelog.md`：记录已经确认进入公式版本变化的内容、原因和影响。
  - `formula-v1-change-notes.md`：记录 v1 草案已经写入、暂未写入、v2 待纳入和编译/检查状态。
  - `current-checkpoint.md`：记录当前讨论位置、最新断点、下一步建议和刚确认但尚未全面沉淀的事项。
  - `collaboration-guidelines.md`：记录协作方法、固定流程、防遗漏机制和用户偏好。
- 如果某条内容一时不确定应进入哪个文件，应先进入 `current-checkpoint.md` 或 `pending-verification-list.md`，后续整理时再归档，不能因为分类犹豫而漏记。

## “入规”快捷约定

- 用户说“入规”时，表示相关内容需要进入 `collaboration-guidelines.md`，并在后续作为固定协作规则执行。
- 收到“入规”后，助手应：
  1. 判断该内容是否适合作为长期协作规则；
  2. 如果适合，写入 `collaboration-guidelines.md`；
  3. 简短告诉用户已经入规，以及写入的大意。
- “入规”只是快捷触发词，不是唯一触发条件。即使用户没有说“入规”，助手也必须根据此前约定主动判断：
  - 是否应写入协作规范；
  - 是否应进入讨论摘要；
  - 是否应进入待验证清单；
  - 是否应进入版本记录；
  - 是否应进入当前断点。
- 不能因为用户没有明确说“入规”，就忽略明显应沉淀为规则、流程或项目记录的内容。

## “记锚”快捷约定

- `入规` = 变成协作规则，写入 `collaboration-guidelines.md`，后续作为固定工作方式执行。
- `记锚` = 变成长期上下文锚点，不一定是协作规则，但需要被保存，方便后续恢复项目状态、理解用户偏好、记住关键背景或回到某个未完成问题。
- 用户说“记锚”时，助手应：
  1. 判断该信息应保存到哪个项目文件；
  2. 优先写入 `current-checkpoint.md`、`formula-discussion-summary.md`、`pending-verification-list.md`，必要时可建议新建长期锚点文件；
  3. 简短告诉用户已记到哪里。
- 适合“记锚”的内容包括：
  - 用户长期偏好；
  - 重要公式设计原则；
  - 后续必须记得的背景；
  - 暂时不改但以后要回来的问题；
  - 用户认为助手以后容易忘的判断。
- `记锚` 不是唯一触发条件。即使用户没有说“记锚”，助手也应主动判断某些信息是否应作为长期上下文保存。

## “退出 / 回来了 / 转新对话 / 新建会话”暗号

- 用户说 `退出` 时，表示用户准备休息或离开。
- 收到 `退出` 后，助手必须停止继续推进新公式内容，改为进行阶段总结和整理。
- 收到 `退出` 或 `新建会话` 时，不能停止后台下载进程、行情缓存进程或其他仍在安全运行的数据任务；只能巡检并记录进度。除非用户明确要求停止，或进程出现破坏性异常，否则必须保持下载继续。
- 阶段总结至少包括：
  - 当前主线；
  - 已确认结论；
  - 当前未完成的小节；
  - 下一步应该从哪里继续；
  - 已同步写入哪些项目文件；
  - 如有必要，补充更新 `current-checkpoint.md`。
- 用户说 `回来了` 时，表示从上次断点继续。
- 收到 `回来了` 后，助手应先读取或参考 `current-checkpoint.md` 的最新断点，简短恢复上下文，然后继续上次未完成的小节。
- 用户说 `转新对话` 或 `新建会话` 时，表示准备把当前项目切换到一个新对话继续，目的是降低单一会话过长导致的反应变慢、遗漏或上下文混乱。
- 收到 `转新对话` 或 `新建会话` 后，助手必须：
  1. 停止继续推进新公式或新功能；
  2. 进行全面交接总结，至少包括当前主线、已完成事项、未完成断点、下一步动作、当前风险点、不要混用的旧分支、最新可用文件；
  3. 更新必要记录文件，优先包括 `current-checkpoint.md`、当前主线专用 checkpoint 和 `new-chat-handoff.md`，必要时同步 `collaboration-guidelines.md`、`pending-verification-list.md`、文件索引或相关计划文件；
  4. 明确告诉用户新会话需要先读取哪些文件、哪些文件只是按需参考；
  5. 把这些文件的可点击链接和完整本地路径逐条发给用户，避免新对话找不到文件；
  6. 给出一段可直接发给新会话的开场提示词，让新会话能接着本会话继续推进任务。
- 用户补充确认：以后只要用户说“新建会话”，助手就自动执行上述全面总结与路径清单流程，不需要用户再次说明“为什么新建”或“要总结什么”。该流程的目标是让新会话接着本会话继续推进任务，而不是重新解释项目。
- 用户补充确认：新会话交接必须要求新会话“全面读取”必读文件，不能只给 2-3 个主线文件，也不能用简短摘要替代协作规范。必读清单至少包括：GitHub 交接包 `manifest.md / handoff.md / required-paths.md`，本地 `current-checkpoint.md`、`next-session-resume.md`、`collaboration-guidelines.md`、`long-term-anchors.md`、当前主线专用记录文件，以及本轮交接包内标注的其他必读文件。
- 新会话启动提示词必须显式写出：`请完整读取以下文件，再恢复上下文`。如果因工具限制无法一次完整读取，必须在新会话中明确说明哪些文件未完整读取，并优先补读，不能在未读完协作规范和长期锚点的情况下继续推进关键决策。
- 用户补充确认：以后用户说 `退出` 或 `新建会话` 时，除了总结、整理记录、列路径，还要把本次交接相关文件同步到 GitHub 的专用文件夹，方便用户在手机端继续访问和推进任务。
- GitHub 专用文件夹默认使用仓库内 `github-mobile-sync/`。同步内容包括最新交接总结、必读路径清单、当前断点、协作规范、主线专用 checkpoint、相关公式文本、验证计划和小型报告；不上传 `data/raw`、虚拟环境、临时缓存、日志大文件、账号密钥或不适合公开/同步的内容。
- GitHub 同步必须按会话/主线隔离，不能把不同会话的交接包混在同一个目录里。当前 B 对话卖点 v13 专属目录固定为 `github-mobile-sync/b-sell-v13/`；A 对话或其他主线必须使用自己的专属目录。除非用户明确要求迁移或合并，助手不得把 B 目录内容写入 A 目录，也不得用 A 目录内容覆盖 B 目录。
- GitHub 同步包必须采用不可覆盖快照：交接总结放入 `github-mobile-sync/b-sell-v13/handoffs/YYYYMMDD-HHMMSS/`，手机端推进记录放入 `github-mobile-sync/b-sell-v13/mobile-updates/YYYYMMDD-HHMMSS/`。不得复用旧时间戳目录，不得用同名文件覆盖旧交接包；如需维护索引，只能追加或清楚标明新旧版本，不能删除历史上下文。
- 每次准备上传 GitHub 前，助手必须先做范围核对：列出将同步的文件清单，确认它们都属于当前主线专属目录或明确允许同步的项目文件；发现跨会话、跨主线、同名覆盖或路径不确定时，必须先停下来说明风险，不能继续上传。
- 如果当前仓库尚未配置 GitHub 远端，助手应先生成或更新本地 `github-mobile-sync/` 交接包，并明确说明“尚未配置远端，暂未上传”；配置好远端后再执行推送。
- 用户补充确认：缺少 GitHub 远端配置也属于“缺什么补什么”。后续若发现本地仓库没有 remote，助手应主动检查可用 GitHub 仓库并配置远端；如果只有一个明确可写目标，可直接设为 `origin`，如果存在多个候选或目标明显不确定，必须先避免误推并提示需要选定目标。
- 手机端如果推进了项目，也必须把进展同步回 GitHub 专用文件夹，至少更新交接总结、当前断点、修改文件清单和下一步动作。
- GitHub 同步必须防混淆、防覆盖：不同会话、不同任务、A/B 对话主线不能混放在同一个无区分文件里。每条主线先进入自己的专属目录，再创建时间戳快照，并写入 `manifest.md` 说明来源、任务、分支、包含文件、下一步和不可覆盖事项；B 对话卖点 v13 只使用 `github-mobile-sync/b-sell-v13/`。
- 禁止覆盖旧交接包；同一天同一任务多次同步也必须使用新的时间戳目录或明确版本后缀。除非用户明确要求整理归档，否则不要修改旧同步包内容。若需要一个“最新入口”，只能维护索引文件指向最新快照，不能把历史快照覆盖成最新内容。
- 从 GitHub 拉回电脑时，先读当前主线专属目录下的 `manifest.md` 判断远端进展属于哪个会话/任务，再决定如何合并。不能让手机端旧文件覆盖电脑本地最新文件，也不能把 B 对话卖点公式的进展误合并到 A 对话买点公式主线。
- 从手机切回电脑本地继续时，助手在恢复工作前必须先检查 GitHub 是否有更新；若远端有新内容，先拉取/读取/对比，再恢复本地上下文，避免在旧基础上继续推进。
- 检查远端更新时必须使用安全流程：先 `git fetch` 或读取远端状态，再比较差异；不能用 `reset --hard`、强制覆盖或破坏本地未提交改动。若本地和远端同时改了同一文件，先说明冲突并做合并判断。
- `退出/回来了/转新对话/新建会话` 是协作暗号，不需要用户额外解释原因。

## “自想”暗号

- 用户说 `自想` 时，表示助手需要反思：为什么刚才没有自己提前想到这个盲点、边界或优化点。
- 收到 `自想` 后，助手不能只解释当前公式结论，而要回答“以后如何才能自己想到”。
- 回答至少包括：
  - 这次漏掉的思维环节是什么；
  - 应该新增哪类固定检查；
  - 下次遇到什么触发条件时要自动启动该检查；
  - 是否需要写入协作规范、当前断点、验证计划或长期锚点。
- 如果 `自想` 暴露出的是通用协作方法，应主动判断是否入规；如果是长期偏好或重要背景，应主动判断是否记锚。
- `自想` 的目标不是道歉，而是把一次遗漏转化为后续可执行的思考触发器。

## “你定”快捷语

- 用户说 `你定` 时，表示用户授权助手按照助手当前给出的建议继续推进。
- 收到 `你定` 后，助手不应继续反复要求用户确认同一个低风险选择，而应直接按自己的专业判断推进下一步。
- 推进时仍要保持协作透明：说明采用了哪个方案、为什么采用、对应状态是 `已确认`、`暂不改待验证`、`只进验证副图`、`记录观察` 还是 `不采用`。
- 如果该选择涉及高风险分支、不可逆操作、明显价值偏好、资金交易决策、外部付费、账号密钥、系统级改动或可能破坏用户环境，`你定` 不等于无条件执行；助手必须先说明风险边界并再次征求明确授权。
- 在公式讨论里，`你定` 默认含义是：采用助手当前推荐的公式口径、阈值或讨论顺序，继续推进到下一小节，并同步记录验证副图需要观察的字段。

## “你持续定”快捷语

- 用户说 `你持续定` 时，表示进入连续授权状态：助手可以持续按照自己的当前最佳建议往下推进，先不用在每一个低风险小节点再次征求用户意见。
- 连续推进期间，助手仍要保持过程透明：每完成一个实质小步，要简短说明采用了什么、为什么、状态标签是什么，并同步必要的项目文件。
- `你持续定` 适用于公式口径讨论、低风险公式文本修改、验证字段整理、记录文件同步、非破坏性自检和下一小节连续推进。
- 用户说 `停` 时，连续授权立即结束。助手必须停下来，不再继续推进新内容，先看用户有什么问题、质疑或方向调整。
- `停` 之后不再默认处于 `你持续定` 状态；除非用户再次明确说 `你持续定`，否则恢复普通逐步协作或单次 `你定`。
- 连续授权期间，助手不能把“下一步建议”“阶段总结”“测试通过”“记录已同步”当作自动停止点；这些只能作为过程汇报，随后应继续执行自己刚给出的下一步。
- 连续授权期间，助手只有在以下情况才应停止继续推进：用户明确说 `停`；遇到高风险或需要授权的操作；遇到工具/系统/上下文长度硬限制；继续推进会明显偏离主线、需要用户价值判断或可能混用 A/B 对话主线。
- 如果因为硬限制必须停，助手必须明确说明“这是硬限制停，不是我自行停止”，并给出恢复后应立即继续的具体动作。
- 如果连续推进中遇到高风险、不可逆、真实交易执行、资金操作、账号密钥、外部付费、系统级改动、可能破坏用户环境，或者明显价值偏好选择，`你持续定` 不等于无条件执行；助手必须先说明风险边界并请求明确授权。
- 如果助手发现继续推进会明显偏离当前主线、需要用户价值判断、或可能把 A/B 对话主线混用，必须主动停下说明，不借 `你持续定` 强行推进。

## 规范到执行闭环

- 不能只把原则写进 `collaboration-guidelines.md`，还要检查对应的执行文件是否同步更新。
- 每新增一条协作规范，助手必须判断它是否需要落到以下一种或多种执行载体：
  - `validation-indicator-plan.md`：如果规则影响验证副图字段、诊断链、显示区设计；
  - `pending-verification-list.md`：如果规则影响后续验证、回测、专项观察；
  - `formula-changelog.md`：如果规则对应已确认进入公式版本的变化；
  - `formula-v1-change-notes.md`：如果规则影响 v1/v2 草案状态、暂未写入项或后续版本；
  - `current-checkpoint.md`：如果规则影响当前断点、下一步动作或刚确认但尚未完整归档的事项。
- 每次写入规范后，助手要做一次“执行落点检查”：
  1. 这条规则有没有对应的具体执行文件；
  2. 执行文件里有没有具体字段、步骤或检查项；
  3. 如果没有，要补上，不能只停留在原则层。
- 如果发现“规范有，但执行计划不够硬”，应立即补执行文件，并说明补到了哪里。
- 验证副图字段设计阶段宁可多做字段审计，也不能粗略跳过。因为字段漏掉会直接导致后续无法快速判断某根K线为什么出信号、为什么不出信号，或者信号是否合理。
- 每完成一个验证副图区块，都要主动检查：
  - 是否覆盖底层变量；
  - 是否覆盖动态阈值；
  - 是否覆盖源头、过滤、保护、显示优先级；
  - 是否覆盖新旧公式差异；
  - 是否覆盖后续参数微调需要依赖的中间变量。
## 文件核对必须用行号定位

当用户反馈“文件内容不对、开头结尾不一致、复制的代码不像你说的那样”时，不能凭印象判断。

必须执行：

- 先核对文件真实头部、关键变量位置、关键分区位置、尾部；
- 用具体行号说明，例如 `PCTB` 在第几行、验证副图区块从第几行开始；
- 不把“文件包含某段内容”误说成“文件开头就是某段内容”；
- 如果用户截图和我描述不一致，优先相信截图和文件核对结果，立即修正判断。

## 东方财富通验证副图采用“算完马上画”

东方财富通对后置绘图不稳定。验证副图不能采用“前面全部计算、最后统一 STICKLINE/DRAWTEXT”的结构。

必须执行：

- 先计算一小组变量；
- 立刻输出这一组对应的线、柱、文字；
- 再继续计算下一组变量；
- 再立刻输出下一组对应显示；
- 输出项尽量靠前，避免公式前段被判定为“没有输出项”；
- 若某模块变量天然很靠后，优先拆成更小副图或状态码图，而不是硬塞进超长后置绘图。

## 长上下文切换与交接

本项目属于长期公式项目，不能把全部上下文压力都压在单一聊天记录里。

当出现以下情况时，助手应主动建议新开对话，而不是继续硬撑：

- 当前对话上下文已经明显过长；
- 讨论已经从普通聊天变成长期项目协作；
- 助手开始出现卡顿、遗漏、反应变慢、重复犯刚确认过的规则；
- 用户开始频繁提醒“你又漏了”“你刚刚才说过”“你需要回顾”；
- 项目文件已经能承载主要上下文，新对话可通过文件恢复。

建议新开对话时，助手必须同时准备或更新交接材料，至少包括：

- 当前主线；
- 已确认结论；
- 当前最新版本文件；
- 待继续的小节；
- 必读项目文件；
- 不应继续考虑的旧分支；
- 新对话的建议开场提示词。

执行原则：

- 新开对话不是中断项目，而是降低长上下文带来的质量损耗；
- 新对话开始后，优先读取项目文件恢复状态，而不是要求用户重新解释全部历史；
- 如果已经存在 `new-chat-handoff.md`，应优先更新它；
- 如果没有交接文件，应主动创建；
- 交接文件必须以当前最新确认状态为准，不能把旧结论误当成当前结论。
- 以后只要 `current-checkpoint.md`、当前主线、当前可用视图、下一步建议或协作规则发生了足以影响新会话恢复的变化，助手都应主动更新 `new-chat-handoff.md`，不等待用户提醒。
## 工具和环境自修复规则

- 后续遇到工具找不到、命令不可用、依赖缺失、PATH 未刷新、版本冲突等环境问题时，助手应先主动排查并尽量自行修复，不因为工具暂时不可用而降低处理质量。
- 用户确认入规：处理工具和环境缺口时遵循“缺什么补什么”，以高质量、高效率为优先。不能因为当前命令不可用就停在低质量方案；应主动寻找等价高质量路径，例如换用兼容命令、补脚本、补依赖、使用项目内已有工具或直接创建更稳的自动化。
- 排查顺序：先检查 `PATH`、命令别名、安装位置、版本冲突；再检查项目内是否已有可用替代；最后再决定是否安装缺失依赖。
- 能用现有环境解决的，助手应直接处理，例如确认并使用 `bash`、`node`、`npm`、`python`、`pip`、`git` 等已安装工具。
- 缺少项目依赖时，优先采用项目内或局部安装方式，例如 `npm install`、Python 虚拟环境、局部工具包、项目脚本依赖；安装后必须验证版本和可用性，再继续推进。
- 不应因为当前找不到某个工具，就退回低质量方案、手工绕路或减少验证步骤；能补环境就先补环境，再完整推进任务。
- 只有涉及账号密钥、付费服务、很大的全量数据下载、系统级高风险改动、可能破坏用户环境的操作时，才需要停下来说明边界并请用户确认。

## 后台任务和本地服务状态巡检

- 只要当前项目存在后台下载、数据缓存、本地服务、长时间脚本、监听端口等“静默运行”的任务，助手在继续推进主线时应顺手检查其状态。

- 默认检查内容包括：
  - 后台进程是否仍在运行；
  - 日志尾部是否还在推进；
  - 已生成文件或缓存数量是否增长；
  - 本地服务端口是否仍在监听；
  - 如有错误日志，是否出现需要立刻处理的失败。
- 如果任务仍正常推进，只简短同步进度，不展开无关细节。
- 如果任务停止、报错或进度异常，应先判断是否能安全恢复；能恢复则恢复并说明，不能恢复则停下来说明原因和下一步选择。

## 缺口自动补齐规则

- 用户确认入规：后续执行一律遵循“缺什么补什么，以高质量和高效为原则”。发现缺少必要工具、依赖、脚本、数据、缓存、索引、远端配置或辅助说明文件时，不停留在“记录缺口”或“低质量绕路”层面，而应优先补齐后再继续推进主线。

- 用户确认入规：后续推进验证、公式讨论、实验环境或数据分析时，若发现缺少必要的 skill、软件、依赖、脚本、数据、缓存或索引环境，不再只记录为缺口或停在建议层，而应优先补齐后继续推进。
- 补齐缺口的标准不是“能凑合完成”，而是“在当前约束下高质量、高效率完成”：优先选择可复用、可验证、可记录、可自动化的方案；临时手工绕路只能作为短期兜底，不能替代应该补齐的能力。
- 执行顺序：先确认项目内是否已有可复用能力；没有则补脚本/测试/依赖/数据；补完后运行最小验证；再把新增能力和路径同步到 `current-checkpoint.md`、`new-chat-handoff.md`、索引文件和相关验证计划。
- 这条规则不覆盖系统安全边界：联网下载、安装依赖、系统级修改、付费服务、账号密钥、高风险或不可逆操作，仍按工具权限要求申请授权；授权后继续补齐，不把授权流程当作普通停止点。
- 对公式项目而言，若验证结论依赖市场环境、指数、行业、成交额、缓存或实验室视图，而本地没有对应数据或工具，应先补对应数据/工具，再继续判断。
