# 长期锚点

本文件记录需要在后续对话、公式整理、验证副图、回测脚本中长期记住的项目锚点。它不是某一次讨论的临时结论，而是后续工作默认要遵守的背景。

## 主公式与验证副图必须同步

用户确认的长期锚点：

> 后续我对主公式做任何调整的时候，你都想到如何调整所有验证副图的公式，以便时时让副图和主公式对应。

执行含义：

- 每次主公式条件、变量名、阈值、窗口、显示优先级发生变化时，都要同步检查验证副图。
- 不允许出现“主公式已改，副图仍按旧口径解释”的情况。
- 需要同步检查的范围包括：
  - A/B/C/D 验证副图；
  - 所有当前仍被采用、有效、最新定稿或准定稿的验证副图版本，不限于 `mini/debug/readable` 这几个命名；
  - 回测字段；
  - 新旧公式对比表；
  - 待验证清单；
  - 版本记录和当前断点。
- 如果某个副图暂时不更新，必须明确标注“该副图口径落后于主公式，不可用于最终判断”。

后续固定检查句：

```text
这次主公式调整会影响哪些验证副图？
对应副图是否需要同步改变量、阈值、文字、颜色或读法？
```

## 讨论过程中同步整理验证副图信息

用户确认的长期锚点：

> 希望和助手讨论过程中，助手同时整理验证副图的必要信息。

执行含义：

- 讨论主公式时，不等公式定稿后才补验证副图字段，而是在每个小模块讨论过程中同步整理。
- 每确认或暂定一个条件，都要顺手记录它在验证副图中需要显示什么：
  - 底层变量；
  - 窗口长度；
  - 阈值或动态阈值；
  - 条件是否成立；
  - 冷却、过滤、互斥或优先级影响；
  - 显示文字、颜色和读法；
  - 后续需要验证的副作用。
- 如果某项还没确认，也可以先记录为“待确认字段”，但不能混成已确认口径。
- 后续整理主公式版本时，应能从讨论记录直接反推出验证副图需要哪些字段，而不是重新回忆。

后续固定检查句：

```text
这一小块需要在验证副图里显示哪些变量、阈值和状态？
哪些字段已确认，哪些只是待确认？
```

## 行情数据必须本地留存

用户确认的长期锚点：

> 记得把股票行情数据保存下来，免得以后接口不能用了。

执行含义：

- 本地公式验证实验室通过 AKShare、Tushare 或其他接口取得的核心行情数据，必须优先落盘缓存。
- 至少保存公式验证需要的核心字段：日期、开高低收、成交量、成交额、涨跌幅、换手率等可取得字段。
- 复权数据必须记录复权口径，当前买点验证主线默认使用前复权 `qfq` 数据。
- 缓存数据要记录股票代码、时间范围、数据源、下载时间和复权方式，避免以后不知道数据从哪里来、能否复现。
- 已下载并用于验证的样本数据不要随手删除；如果后续清理空间，要先确认是否已有备份或可重新下载。
- 大规模全 A 下载前仍需控制范围和节奏，避免一次性下载过大；但只要下载了，就尽量保存为本地可复用数据包。
- 数据缓存可以继续放在 Git 忽略目录里，不需要提交进代码仓库；关键是本机要能复用，且记录清楚口径。

后续固定检查句：

```text
这次验证用到的行情数据是否已经本地保存？
是否记录了代码、时间范围、数据源、复权方式和下载时间？
如果接口失效，当前样本是否还能离线复现？
```

## “自想”暗号：把遗漏转化为固定思考触发器

用户确认的长期锚点：

> 新增暗号：自想，意味着你需要反思如何才能想到这个。

执行含义：

- 当用户说 `自想` 时，助手需要停止只讨论当前结论，转而反思“为什么刚才没有主动想到”。
- 反思重点不是道歉，而是把一次遗漏提炼成下一次可执行的检查机制。
- 回答时至少说明：
  - 这次漏掉的是哪一类思维动作；
  - 如果从第一性原理、边界样本、互斥关系、过程结构、基准 K、窗口口径等角度检查，本来应该如何想到；
  - 后续遇到什么触发条件时，应自动启动对应检查；
  - 是否需要写入协作规范、当前断点、验证计划或公式摘要。
- 如果 `自想` 触发的是通用方法，要优先沉淀到 `collaboration-guidelines.md`；如果是长期项目偏好，要同时记到本文件或当前断点。

后续固定检查句：

```text
这次我为什么没有自己提前想到？
它属于哪类可复用检查：第一性原理、互斥边界、过程结构、基准K、窗口参照、阈值边界，还是显示层互斥？
下次遇到什么信号或语境时，我要自动启动这个检查？
```

## “你定”暗号：授权助手按建议推进

用户确认的长期锚点：

> 当用户说 `你定` 时，助手按自己当前建议继续推进。

执行含义：

- `你定` 表示用户不想在低风险细节上继续逐项确认，授权助手采用当时最稳妥的建议。
- 助手应直接推进下一步，并说明采用了哪个建议、为什么采用、当前状态标签是什么。
- 公式讨论中，`你定` 默认允许助手继续按建议确定口径、阈值或讨论顺序，并同步整理验证副图字段。
- 如果涉及高风险、不可逆、交易执行、系统级改动、外部付费、账号密钥或明显价值偏好，仍需单独说明风险并请求明确授权。

后续固定检查句：

```text
用户说“你定”后，我是否已经按当前最佳建议推进，而不是重复追问？
这个选择是否低风险、可回退、属于公式讨论口径？如果不是，是否需要额外说明风险并确认？
```

## “你持续定”暗号：连续授权推进

用户确认的长期锚点：

> 当用户说 `你持续定` 时，助手可以持续按照自己的建议推进，直到用户说 `停`。

执行含义：

- `你持续定` 表示用户暂时不想逐项确认，授权助手连续处理低风险的公式讨论、公式文本修改、验证字段整理、记录文件同步和自检。
- 助手应持续按当前最佳判断推进，每完成一个实质小步要简短说明采用了什么、为什么、状态标签是什么。
- 用户说 `停` 时，连续授权立即结束；助手必须停止继续推进新内容，先回答用户的问题或等待新的方向。
- `停` 之后不再默认继续 `你持续定`，除非用户再次明确授权。
- 涉及高风险、不可逆、真实交易执行、资金操作、账号密钥、外部付费、系统级改动、可能破坏用户环境或明显价值偏好的选择时，仍需单独说明风险并请求明确授权。

后续固定检查句：

```text
当前是否处于“你持续定”连续授权状态？
用户是否已经说“停”？如果说了，我是否已经停止继续推进新内容？
当前动作是否属于低风险、可回退、公式讨论/记录/自检范围？如果不是，是否需要额外确认？
```

## GitHub 手机同步必须防混淆、防覆盖

用户确认的长期锚点：

> 上传的文件不要跟其他会话弄混了，更加不要进行覆盖。

执行含义：

- GitHub 手机同步不能把不同会话、不同任务、A/B 对话主线混放在同一个无区分文件里。
- 每条主线必须先进入自己的专属命名空间目录，再创建时间戳快照；当前 B 对话卖点 v13 专属目录固定为 `github-mobile-sync/b-sell-v13/`。
- 每次 `退出`、`新建会话` 或手机端推进后同步，都必须创建独立交接包目录，目录名包含日期时间、会话/任务标签和本地分支信息。
- 当前 B 对话默认目录格式：

```text
github-mobile-sync/b-sell-v13/handoffs/YYYYMMDD-HHMMSS/
github-mobile-sync/b-sell-v13/mobile-updates/YYYYMMDD-HHMMSS/
```

- 每个交接包必须包含清单文件，例如 `manifest.md`，写清楚：
  - 本次同步属于哪个会话/任务；
  - 来源是电脑本地还是手机端；
  - 基于哪个本地分支或 GitHub 分支；
  - 包含哪些文件；
  - 下一步要从哪里继续；
  - 哪些文件只是参考，不能覆盖当前主线。
- 禁止直接覆盖旧交接包。若同一天同一任务多次同步，也要生成新的时间戳目录，或显式追加版本后缀。
- 从 GitHub 拉回电脑时，必须先对比 manifest，再决定哪些内容合并到本地；不能用手机端旧文件覆盖电脑本地最新文件。
- 如果发现远端和本地同名文件都有改动，先判断是否同一会话、同一主线、同一时间顺序；无法确认时必须停下说明冲突，不自动覆盖。
- 如果需要维护“最新入口”，只能用索引文件指向最新快照；不能把旧快照目录里的文件覆盖成最新内容。

后续固定检查句：

```text
这次 GitHub 同步包是否有独立目录和 manifest？
它属于哪个会话/任务，是否会和其他会话混淆？
有没有任何文件会覆盖旧交接包或覆盖本地最新主线？
它是否写进了当前主线专属目录，而不是误写到其他会话目录？
```
